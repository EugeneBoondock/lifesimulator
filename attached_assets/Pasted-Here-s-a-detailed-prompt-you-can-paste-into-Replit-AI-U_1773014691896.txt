**Prompt for Replit AI**

I am building a life simulator in `three.js`. Right now the characters look bad, too primitive, too static, and they do not perform believable actions. I want you to refactor the character system so the simulation feels like a real little AI society instead of a bunch of placeholder dolls.

Your job is to improve this in a production-minded way, not with hacks. Keep the current game running, but replace the current character pipeline with a cleaner, scalable one.

## Main goals

1. Replace the current ugly low-detail “toy” characters with better stylized 3D characters that still fit the world’s soft low-poly look.
2. Give every AI a proper body structure and readable silhouette.
3. Add animations and visible actions so agents do not look dead.
4. Make the system scalable for many agents at once.
5. Preserve performance in browser.
6. Keep everything compatible with `three.js`, not React Three Fiber unless absolutely needed.

## Visual direction

The world has a charming soft, low-poly, peaceful simulator vibe. The characters should match that style.

I do **not** want:

* random cubes stuck together
* tiny stiff puppet bodies
* characters with no joints
* dead-eyed NPCs standing frozen
* flat placeholder mannequins
* stiff movement with zero anticipation or follow-through

I **do** want:

* stylized low-poly humanoids
* proper proportions with head, torso, pelvis, upper/lower arms, upper/lower legs, hands, feet
* slightly oversized heads for charm and readability
* simple but attractive facial features
* variation in outfit colors, hair, accessories, body type, height, and posture
* a clean, game-like art style similar to cozy simulation games
* readable silhouettes even from a distance

## Character redesign requirements

Create a modular stylized humanoid system.

Each character should have:

* head
* hair or hat slot
* torso
* pelvis/hips
* upper arms
* lower arms
* hands
* upper legs
* lower legs
* feet
* optional backpack/tool slot
* optional accessory slot

Do not just stretch cubes. Shape the mesh so it feels intentionally designed.

### Proportion rules

* Head slightly larger than realistic for personality
* Torso compact but not boxy
* Arms and legs should taper slightly
* Feet should be a little larger than realistic for charm and grounding
* Hands can be simplified but should exist
* Neck should exist, even if subtle
* Shoulder width should look natural
* Hips should visually support walking animation

### Surface style

* low-poly but clean
* smooth color blocking
* minimal texture dependence
* use flat or softly shaded materials
* avoid noisy textures
* avoid photo realism
* keep materials readable in daylight

## Strong recommendation for asset pipeline

Do **not** keep building characters entirely from primitive geometry unless used only for quick fallback prototypes.

Preferred pipeline:

1. Create or source 1–3 good stylized low-poly base characters.
2. Export them as `.glb`.
3. Load them with `GLTFLoader`.
4. Rig them with a humanoid skeleton.
5. Attach animation clips.
6. Clone them for multiple agents using a safe clone method for skinned meshes.
7. Randomize outfit colors, accessories, scale, and idle variants per agent.

If needed, use a proper DCC workflow:

* Blender for authoring characters
* Mixamo for quick auto-rigging and basic animation clips
* export as `.glb` for `three.js`

If you cannot make new custom art immediately, then create a better procedural character generator inside `three.js` with actual limb hierarchy and joints, but structure it so it can later swap to `.glb` characters without rewriting the AI logic.

## Rigging and animation requirements

Every agent must support a real animation state machine.

Minimum animation states:

* idle
* lookAround
* think
* walk
* jog or hurry
* gather
* eat
* talk
* wave
* sit
* sleep
* inspect
* celebrate
* sad or low-energy idle
* hurt or exhausted idle
* interactWithObject

If some animations do not exist yet, create placeholders using procedural joint animation until imported clips are ready.

### Animation behavior

Agents must never stand completely still unless asleep or intentionally frozen.

Idle should include tiny motions:

* breathing
* slight sway
* head turn
* blinking if supported
* weight shifting
* arm settling
* occasional looking at nearby objects or agents

Walking must include:

* arm swing
* hip motion
* foot lift
* slight torso counter-rotation
* natural cadence variation

Eating must include:

* bend or squat if near ground food
* hand-to-mouth or lean-to-object motion
* short chew loop
* satisfied return-to-idle motion

Talking must include:

* facing target
* head nods
* subtle hand gestures
* occasional shift in posture

Thinking or observing must include:

* pause
* head tilt
* eyes or head tracking if possible
* small body stillness with occasional motion

## AI-to-animation bridge

The animation system should be driven by AI state, not random clip spam.

Create a clean mapping layer like:

* `needs.food.high` → search food → walk → gather → eat
* `social.intent.talk` → approach target → stop → face target → talk animation
* `energy.low` → slow walk or exhausted idle
* `task.gathering` → gather animation at resource node
* `mood.happy` → lighter idle, occasional hop or cheerful gesture
* `mood.sad` → slower movement, drooped posture
* `threat.detected` → alert pose, faster movement
* `sleeping` → sleep pose and no idle chatter

I want the simulation to look like characters are acting because of internal states.

## Architectural refactor

Separate the system into these layers:

### 1. `CharacterAssetManager`

Responsible for:

* loading base character `.glb` files
* loading animation clips
* caching assets
* optional Draco support
* returning reusable references

### 2. `CharacterFactory`

Responsible for:

* creating a new agent visual from a base character
* applying random variation
* assigning materials and colors
* attaching name label anchor points
* setting scale and offsets

### 3. `CharacterAnimator`

Responsible for:

* creating an `AnimationMixer` per character
* managing actions
* crossfading between clips
* controlling playback speed
* exposing methods like `playState('walk')`, `playState('eat')`, `setMood('happy')`

### 4. `CharacterController`

Responsible for:

* position updates
* target following
* turning
* movement smoothing
* path steering
* syncing movement speed to animation speed

### 5. `CharacterBrainAdapter`

Responsible for:

* listening to AI needs, mood, job, social status
* converting simulation state into animation and posture requests

Do not leave everything inside one giant scene update loop.

## Movement quality

The current movement probably looks robotic. Fix that.

Requirements:

* smooth turning before walking
* no instant snap rotation unless explicitly needed
* acceleration and deceleration
* stop with slight settle motion
* optional footstep timing events
* different movement styles based on urgency
* no sliding across the ground while idle animation plays

Animation speed should scale with actual movement speed so walk cycles match distance traveled.

## Interaction quality

Characters should visibly react to world events.

Examples:

* see mushroom → look at it first
* move toward it
* crouch or bend to collect
* lift object or simulate pickup
* return to neutral
* if hungry, eat soon after
* if another agent approaches, turn to face them
* if weather changes, maybe react
* if tired, posture softens
* if idle too long, self-directed micro actions begin

## Social believability

Make AI characters feel aware of one another.

* small glance away and back

## Performance rules

The simulator may have many agents, so do not destroy performance.

Do this:

* use proper asset reuse
* keep one shared base character asset and clone it
* share materials where possible
* use lower-poly LOD for far agents
* reduce animation update frequency for distant agents
* use billboards or very cheap proxies for far-away characters if necessary
* use `InstancedMesh` for repeated props like mushrooms, rocks, trees, signs, but not for unique animated skinned characters unless you deliberately build a custom GPU animation solution
